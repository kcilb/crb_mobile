1. Overview

REST API for member verification and deposit notification. Channels and partners send requests to Quantive Core (QCore) to verify members by account number or to notify QCore of deposits. Use the API to:

  • Confirm an account exists
  • Retrieve member name, account status, and currency
  • Send a deposit notification for posting to a member account

All requests/responses use JSON over HTTPS. Send the bearer token obtained from the Authentication API in the Authorization: Bearer <token> header for all protected endpoints.

3. JWT Authentication

This API uses JWT (JSON Web Token) for authentication. Obtain a bearer token from the Authentication API and use it for all subsequent requests to protected endpoints (e.g. verifyMember, receiveDepositNotification).

3.1 Request for token (payload)

Obtain a bearer token before calling any protected endpoint.

3.2 Using the token

For all subsequent requests to protected endpoints, send the token in the Authorization header and specify Content-Type: application/json. 

Authorization: Bearer <token>
Content-Type: application/json

The request body depends on the endpoint:

  • For verifyMember: requestId and memberAccount only (see 5.1)
  • For receiveDepositNotification: see 5.2 for full payload

Do not send username/password in these protected requests.

3.3 Response

Success responses return endpoint-specific data. HTTP 401/403 indicate missing, invalid, or expired token; obtain a new token (3.1) and retry. Application-level codes are documented in Section 6.

4. Request and Response Format

Requests use POST with Content-Type: application/json. Response bodies are JSON with standard fields and endpoint-specific data.

Standard response fields:

  • responseCode – application-level code (see Section 6)
  • requestId – echo of the request or header requestId
  • responseMessage – human-readable message

6. Response Codes

Application-level response codes used by this API. These are aligned with the Card Integration API response codes.

CodeDescription
0Success
400Bad Request — Invalid or malformed request payload
401Unauthorized — Missing or invalid authentication
403Forbidden — Access not allowed for this resource
404Not Found — Resource or endpoint not found
422Unprocessable Entity — Validation error (e.g. invalid account number)
500Internal Server Error — Server-side error; retry or contact support
-98Validation error — Required field missing or invalid
-99Business / general error — See responseMessage for details

7. Error Handling

Always check responseCode before using response data (see Section 6). HTTP status and/or body may carry 400, 401, 403, 404, or 500; application codes 0, -98, and -99 are in responseCode. Use requestId for correlation and support.
